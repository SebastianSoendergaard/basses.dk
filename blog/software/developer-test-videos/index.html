Developer testing

Below I have colleded a number of videos that in my opinion gives some good insights on how to do testing.

TDD, Where Did It All Go Wrong - by Ian Cooper

Talks about how most do testing and why TDD to some has a bad reputation
Tells how TDD should in fact be practiced

Main takaways:
  • tests that is tighly coupled to implementation will break too often thus becomming frustrating
  • a unit is not a single class ot method, it is a module with a defined interface, that will rarely change
  • the module should be seen as a black box so testing should only be on input and output
  • the internal of the module is implementation details that should be allowed to change frequently
  • test should focus on behaviour instead of data
  • TDD is basically the same as BDD

Building Operable Software with TDD (but not the way you think) - by Martin Thwaites

Suggests testing from the outside in, starting with the public API, either HTTP API or a SDK API

Main takaways:
  • yes tests can include database and other infrastructure calls, and it can still be very fast
  • by testing from the public API you do not couple the tests to implementation or infrastructure, so that can still change
  • endpoint paths, routing, dependency injection are all automatically tested
  • actually you get 100% coverage, if code is not covered, you are either missing a test or you have dead code
  • some stuff can not be verified through API, but by including tracing it can be done
