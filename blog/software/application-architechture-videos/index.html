Application architecture

Below I have colleded a number of videos that in my opinion sums up a lot of the stuff needed for a good application architecture.

Clean Architecture - by Robert C. Martin (Uncle Bob)

Very good talk on the thoughts behind Clean Architectures (including Hexagonal and Onion)
If you know Uncle Bob you can just skip the first 10.5 minutes as they are not relevant at all.

Main takaways:
  • dependencies - make sure they point the right way. Always inwards, your application should never depend on the outside
  • frameworks, databases, internet ect. are just details and you should always decouple your application from it
  • creaming architecture, structure the application so it shows what it does instead of showing it is just another asp.net solution
  • business rules can be devided in two: domain rules, and application rules
    • domain rules are rules that always exists in the business e.g. calculation of interests in a bank, they should be in the domain layer
    • application rules are rules that exists because of the application, they should be in the application layer

Vertical Slice Architecture - by Jimmy Bogard

Jimmy proposes an alternative to the layered architecture by splitting in verticals instead

Main takeaways:
  • prioritize high cohesion over layers
  • do not put all business logic in large horizontal services but split into small feature specific chunks
Personal comments
  • Vertical Slice Architecture works very well as it separates features and use cases into small decoupled chunks. It is posible to implement in parallel. It is easy to test.
  • Just because we structure the files differently the thoughts behind clean architectures are still very relevant, in fact Uncle Bob suggests that the application layer should be split into use cases which is the same as Jimmys Bogards slices. Also see the video below with Derek Comartin for how to combine the thoughts behind both approches.
  • I have seen this architecture beeing used as a excuse for not using DDD and other good principles. That is wrong! Jimmy barely mentiones it in this video but we should still be using domain objects. See his video below on how to refactor a vertical slice handler.

Tired of Layers? Vertical Slice Architecture to the rescue! - by Derek Comartin

Derek suggests how to use a vertical sliced approch still keeping all the good parts from clean architectures in mind

Main takeaways:
  • basically the same as Jimmy Bogard mentiones in his video but in a bit more varied view

Domain-Driven Refactoring - by Jimmy Bogard

See how domian rules are pushed down into the domain objects even when used in Vertical Slice Architecture where each feature is separated from each other

Main takeaways:
  • vertical Slice Architecture is not an excuse to for using rich domain objects that is shared between features
  • vertical Slice Architecture is not an excuse to have a lot of cluttered code with dependencies pointing in all directions
  • how domain logic can be identified and moved to the domain objects
